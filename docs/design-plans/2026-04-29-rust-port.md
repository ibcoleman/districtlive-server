# DistrictLive Rust Port Design

## Summary

DistrictLive is a live music event aggregator for the Washington, DC area. This project ports the existing Kotlin/Spring backend to Rust, preserving the same external API shape, database schema, and feature set. The application ingests event listings from over ten sources — ranging from ticketing APIs (Ticketmaster, Bandsintown, Dice.fm) to venue websites scraped directly (Black Cat, DC9, Comet Ping Pong, and others) — normalizes and deduplicates the results, and serves a public read API alongside a protected admin API. An enrichment pipeline runs separately to fill in artist metadata from MusicBrainz and optionally Spotify.

The port lands directly onto the existing Rust scaffold, which already provides the hexagonal architecture skeleton, Bazel build system, PostgreSQL integration via sqlx, and the Axum HTTP framework. Seven implementation phases build the system bottom-up: database schema first, then domain types and port traits, PostgreSQL adapters, HTTP handlers, the ingestion pipeline and its connectors, the enrichment pipeline, and finally the two TypeScript admin frontend pages. Each phase produces a passing `just check` before the next begins. No new architectural patterns are introduced at the language level — the connectors, enrichers, and background schedulers are all written as port/adapter pairs following the same conventions already established in the scaffold.

## Definition of Done

The Rust version of DistrictLive is complete when:

1. All 21 DB migrations are ported to sqlx format and the full schema is in place (venues, artists, events, sources, ingestion_runs, featured_events + seed data)
2. The public read/write API is working: events list + filters, event detail, venues, artists, featured events — same URL shape as the Kotlin app
3. The ingestion pipeline is ported: all 10+ connectors (Ticketmaster, Bandsintown, Dice.fm, Black Cat, DC9, Comet Ping Pong, Pie Shop, Rhizome DC, Seven Drum City, Union Stage Presents) running as background Tokio tasks behind `INGESTION_ENABLED`
4. The artist enrichment pipeline is ported: MusicBrainz + Spotify enrichers with the `PENDING→DONE` state machine, behind `ENRICHMENT_ENABLED`
5. Admin API is working (`/api/admin/*` routes): source health, ingestion history, manual trigger, featured event management — all protected by HTTP Basic auth
6. The TypeScript frontend has the ingestion monitor + featured events admin pages ported from the Kotlin static assets
7. Fixture-based tests for all scrapers + property tests for normalization/deduplication

## Acceptance Criteria

### rust-port.AC1: DB schema and migrations
- **rust-port.AC1.1 Success:** All 21 migrations apply cleanly against a fresh PostgreSQL database via `just test-integration`
- **rust-port.AC1.2 Success:** All expected tables exist after migration: `users`, `venues`, `artists`, `events`, `event_artists`, `event_sources`, `sources`, `ingestion_runs`, `featured_events`
- **rust-port.AC1.3 Success:** Seed data migrations populate `venues` and `sources` tables with the expected DC venue and source records
- **rust-port.AC1.4 Edge:** Running migrations twice does not error — sqlx migration tracking prevents re-application

### rust-port.AC2: Public read API
- **rust-port.AC2.1 Success:** `GET /api/events` returns a paginated JSON list with correct shape (title, slug, venue, artists, start_time, price, status)
- **rust-port.AC2.2 Success:** `GET /api/events` with `date_from`, `date_to`, `venue_slug`, `genre`, `neighborhood`, `price_max`, and `status` filters each independently narrow results
- **rust-port.AC2.3 Success:** `GET /api/events/{id}` returns full event detail including venue object, artist list, and related events at the same venue within ±7 days
- **rust-port.AC2.4 Success:** `GET /api/venues` with `neighborhood` filter returns only venues in that neighborhood
- **rust-port.AC2.5 Success:** `GET /api/artists` with `local=true` returns only artists flagged as local
- **rust-port.AC2.6 Success:** `GET /api/featured` returns the most recently created featured event with full event detail and blurb
- **rust-port.AC2.7 Failure:** `GET /api/events/{id}` with an unknown UUID returns 404
- **rust-port.AC2.8 Failure:** `GET /api/events` with a malformed `date_from` returns 400

### rust-port.AC3: Ingestion pipeline
- **rust-port.AC3.1 Success:** Each of the 10+ connectors parses its fixture file and produces ≥1 `RawEvent` with non-empty title, venue name, and start time
- **rust-port.AC3.2 Success:** Normalization produces a non-empty URL-safe slug for every `RawEvent`
- **rust-port.AC3.3 Success:** Deduplication groups events with matching slugs into a single `DeduplicatedEvent` with merged source attributions
- **rust-port.AC3.4 Success:** `POST /api/admin/ingest/trigger` when `INGESTION_ENABLED=true` completes and returns stats (fetched, created, updated, deduped)
- **rust-port.AC3.5 Success:** An `IngestionRun` record is written as RUNNING on pipeline start and updated to SUCCESS on completion
- **rust-port.AC3.6 Success:** Running the same connector twice updates the existing event record (slug match → UPDATE, not INSERT duplicate)
- **rust-port.AC3.7 Failure:** Events with placeholder titles (`"private event"`, `"tba"`, `"sold out"`, `"coming soon"`) are dropped before upsert
- **rust-port.AC3.8 Failure:** `POST /api/admin/ingest/trigger` when `INGESTION_ENABLED=false` returns 400 with an explanatory error message

### rust-port.AC4: Artist enrichment pipeline
- **rust-port.AC4.1 Success:** A PENDING artist is claimed, marked IN_PROGRESS, successfully enriched via MusicBrainz, and transitioned to DONE with canonical name, MBID, and tags populated
- **rust-port.AC4.2 Success:** An artist that fails enrichment on every attempt up to `max_attempts` is marked SKIPPED
- **rust-port.AC4.3 Success:** On application startup, any IN_PROGRESS artists left from a previous crashed run are reset to PENDING
- **rust-port.AC4.4 Failure:** A transient enrichment error marks the artist FAILED (not SKIPPED), leaving it eligible for retry on the next batch
- **rust-port.AC4.5 Edge:** Spotify enricher is disabled by default; it only runs when `SPOTIFY_ENABLED=true` in the environment

### rust-port.AC5: Admin API and HTTP Basic auth
- **rust-port.AC5.1 Success:** `GET /api/admin/sources` with valid credentials returns all sources with health fields (last_success_at, consecutive_failures, healthy flag)
- **rust-port.AC5.2 Success:** `GET /api/admin/sources/{id}/history` returns ingestion run history for the specified source, ordered by `started_at` descending
- **rust-port.AC5.3 Success:** `POST /api/admin/featured` with a valid event UUID and non-blank blurb creates and returns the featured event
- **rust-port.AC5.4 Failure:** Any `/api/admin/*` request without an `Authorization` header returns 401
- **rust-port.AC5.5 Failure:** Any `/api/admin/*` request with incorrect credentials returns 401
- **rust-port.AC5.6 Failure:** `POST /api/admin/featured` with a blank blurb returns 400
- **rust-port.AC5.7 Failure:** `POST /api/admin/featured` with an unknown event UUID returns 404

### rust-port.AC6: Frontend pages
- **rust-port.AC6.1 Success:** `pnpm build` in `frontend/` produces bundles for ingestion monitor and featured events pages without TypeScript errors
- **rust-port.AC6.2 Success:** The ingestion monitor page renders source health cards populated from `GET /api/admin/sources`
- **rust-port.AC6.3 Success:** Clicking a manual trigger button calls the ingest endpoint and updates the displayed source status
- **rust-port.AC6.4 Success:** The featured events page renders the history list and successfully submits the create-featured form

### rust-port.AC7: Test coverage
- **rust-port.AC7.1 Success:** All 10+ connector fixture tests pass with `just test`
- **rust-port.AC7.2 Success:** Normalization property tests pass: slug is always non-empty and URL-safe; start_time is always preserved unchanged
- **rust-port.AC7.3 Success:** Deduplication idempotency property test passes: deduplicating an already-deduplicated list produces the same result
- **rust-port.AC7.4 Success:** `just check` passes (cargo fmt + clippy `-D warnings` clean)

## Glossary

- **Axum**: Rust web framework built on Tokio and Tower. Used for all HTTP routing and handler dispatch in this project.
- **Bazel**: Hermetic, language-agnostic build system. Used here via `rules_rust` + `crate_universe` to build the Rust binary and run tests with reproducible sandboxing.
- **`crate_universe`**: Bazel extension (part of `rules_rust`) that reads `Cargo.toml` and generates Bazel build targets for all third-party Rust crates. `just bazel-repin` regenerates its lockfile.
- **Deduplication**: The process of grouping `NormalizedEvent` records that refer to the same real-world event (identified by matching slug) into a single `DeduplicatedEvent` with merged source attributions.
- **`EnrichmentStatus`**: State machine enum for the artist enrichment lifecycle: `PENDING → IN_PROGRESS → DONE / FAILED / SKIPPED`.
- **Fixture-based test**: A test that runs a connector's parsing logic against a saved copy of the external response (an HTML page or JSON payload stored in `tests/fixtures/`) rather than hitting a live network endpoint.
- **Flyway**: Java-ecosystem database migration tool used by the original Kotlin app. Its `V1__` through `V21__` numbered migrations are translated to sqlx's `0001_`–`0021_` format in this port.
- **Hexagonal architecture (ports and adapters)**: Structural pattern where domain logic is isolated from I/O. Traits (`ports/`) define what the domain needs; concrete implementations (`adapters/`) fulfill those traits using databases, HTTP clients, etc. The domain core never imports adapter code.
- **HTTP Basic auth**: Authentication scheme where credentials are sent as a Base64-encoded `Authorization` header. Used here to protect all `/api/admin/*` routes via a Tower middleware layer.
- **`IngestionRun`**: A database record representing one execution of the ingestion pipeline for a source, tracking status (RUNNING / SUCCESS / FAILED), timing, and event counts.
- **JSON-LD**: A format for embedding structured data inside an HTML page as a `<script type="application/ld+json">` tag. The Dice.fm connector extracts event data from this rather than from DOM elements.
- **MusicBrainz**: Open-licensed music metadata database. The enrichment pipeline queries its search API to retrieve canonical artist names, MBIDs, and genre tags.
- **Newtype**: A Rust idiom of wrapping a primitive in a single-field struct (e.g., `VenueId(Uuid)`) to give it a distinct type. Prevents accidentally passing one kind of ID where a different kind is expected.
- **`proptest`**: Rust property-based testing library. Generates large numbers of random inputs to verify invariants (e.g., slug is always non-empty and URL-safe) rather than testing only hand-written examples.
- **`reqwest`**: Async HTTP client library for Rust. Used by all connectors and enrichers to fetch data from external APIs and venue websites.
- **`RawEvent`**: The unprocessed output of a `SourceConnector::fetch()` call — a connector-specific struct with title, venue name, start time, and other fields before normalization.
- **`scraper`**: Rust HTML parsing library with a CSS selector API. Used by the venue-website connectors to extract event data from HTML pages.
- **`SELECT FOR UPDATE SKIP LOCKED`**: A PostgreSQL locking hint that allows multiple concurrent workers to claim distinct rows from a queue without contention. Used in `claim_pending_batch` to safely distribute artist enrichment work across concurrent Tokio tasks.
- **Slug**: A URL-safe identifier derived from an event's title, venue name, and date (e.g., `the-national-9-30-club-2026-05-15`). Used as the deduplication key and as a stable URL segment.
- **sqlx**: Rust library for async PostgreSQL access with compile-time SQL verification. Also provides the migration runner (`sqlx::migrate!()`) that applies numbered `.sql` files at startup.
- **`thiserror`**: Rust library for deriving `std::error::Error` implementations on enums. Used for all domain error types (`RepoError`, `IngestionError`, `EnrichmentError`).
- **`tokio-cron-scheduler`**: Rust library that runs async tasks on a cron schedule inside the Tokio async runtime. Used to trigger ingestion and enrichment pipelines on configured intervals.
- **Tower**: Rust middleware framework underlying Axum. The admin Basic auth check is implemented as a Tower middleware using `middleware::from_fn_with_state`.
- **`UpsertResult`**: Enum returned by `EventRepository::upsert` indicating whether the operation inserted a new row (`Created`) or updated an existing one (`Updated`), derived from PostgreSQL's `xmax` system column.
- **Vite**: Frontend build tool. The multi-entry configuration (`build.rollupOptions.input`) produces separate bundles for the ingestion monitor and featured events admin pages alongside the existing public frontend.
- **`xmax`**: PostgreSQL internal system column. After an `INSERT … ON CONFLICT DO UPDATE`, a non-zero `xmax` indicates the row was updated rather than inserted, giving a reliable way to distinguish `Created` from `Updated` without a second query.

## Architecture

Hexagonal architecture from the Kotlin original is preserved directly onto the existing Rust scaffolding's `domain/`, `ports/`, `adapters/`, and `http/` modules. Ingestion and enrichment pipeline logic lives in top-level `src/ingestion/` and `src/enrichment/` modules; their concrete adapters (connectors, enrichers) live under `src/adapters/`.

```
src/
  domain/                       - Pure value types: Venue, Artist, Event, EventSource,
  │                               Source, IngestionRun, FeaturedEvent; enums; newtypes
  ports/                        - Async trait definitions
  │   venue_repository.rs
  │   artist_repository.rs
  │   event_repository.rs
  │   featured_event_repository.rs
  │   source_repository.rs
  │   ingestion_run_repository.rs
  │   source_connector.rs       - Ingestion port (fetch() → Vec<RawEvent>)
  │   artist_enricher.rs        - Enrichment port (enrich() → Option<EnrichmentResult>)
  adapters/
  │   pg_venues.rs
  │   pg_artists.rs             - Includes claim_pending_batch (SELECT FOR UPDATE SKIP LOCKED)
  │   pg_events.rs              - Includes upsert (resolve venue/artists, ON CONFLICT slug)
  │   pg_featured.rs
  │   pg_sources.rs
  │   pg_ingestion_runs.rs
  │   connectors/               - SourceConnector impls
  │   │   ticketmaster.rs       - reqwest JSON client
  │   │   bandsintown.rs        - reqwest JSON client
  │   │   dice_fm.rs            - reqwest + JSON-LD extraction
  │   │   black_cat.rs          - reqwest + scraper CSS selectors
  │   │   dc9.rs
  │   │   comet_ping_pong.rs
  │   │   pie_shop.rs
  │   │   rhizome_dc.rs
  │   │   seven_drum_city.rs
  │   │   union_stage_presents.rs
  │   enrichers/                - ArtistEnricher impls
  │       musicbrainz.rs        - MusicBrainz name lookup (rate-limited)
  │       spotify.rs            - Spotify name lookup (optional, disabled by default)
  http/
  │   events.rs / venues.rs / artists.rs / featured.rs / admin.rs / version.rs
  │   middleware/
  │   │   basic_auth.rs         - Tower middleware: extract + validate Authorization header
  │   error.rs / mod.rs
  ingestion/
  │   orchestrator.rs           - fetch → normalize → filter → deduplicate → upsert pipeline
  │   normalization.rs          - RawEvent → NormalizedEvent
  │   deduplication.rs          - slug-keyed dedup, source attribution merging
  │   scheduler.rs              - tokio-cron-scheduler wiring, INGESTION_ENABLED gate
  enrichment/
  │   orchestrator.rs           - claim batch → enrich → update state machine
  │   scheduler.rs              - ENRICHMENT_ENABLED gate
  config.rs                     - Config::from_env()
  lib.rs / main.rs
```

`AppState` (in `http/mod.rs`) holds Arc-wrapped adapters (one per port) and `Arc<Config>`. The ingestion and enrichment schedulers are spawned as detached Tokio tasks in `main.rs` after the HTTP server starts, conditional on their respective env flags.

### Key Contracts

**SourceConnector port** (all connector adapters implement this):

```rust
trait SourceConnector: Send + Sync {
    fn source_id(&self) -> &str;
    fn source_type(&self) -> SourceType;
    async fn fetch(&self) -> Result<Vec<RawEvent>, IngestionError>;
    fn health_check(&self) -> bool { true }
}
```

**ArtistEnricher port**:

```rust
trait ArtistEnricher: Send + Sync {
    fn source(&self) -> EnrichmentSource;
    async fn enrich(&self, name: &str) -> Result<Option<EnrichmentResult>, EnrichmentError>;
}
```

**EventRepository upsert** (the most complex adapter method):

```rust
trait EventRepository {
    async fn upsert(&self, cmd: EventUpsertCommand) -> Result<UpsertResult, RepoError>;
    async fn find_all(&self, filters: EventFilters, page: Pagination)
        -> Result<Page<Event>, RepoError>;
    // + find_by_id, find_by_slug, find_upcoming, find_by_venue_id,
    //   count_upcoming_by_venue
}

enum UpsertResult { Created, Updated }

struct EventFilters {
    date_from: Option<OffsetDateTime>,
    date_to: Option<OffsetDateTime>,
    venue_slug: Option<String>,
    genre: Option<String>,
    neighborhood: Option<String>,
    price_max: Option<Decimal>,
    status: EventStatus,
}
```

## Existing Patterns

This design follows the hexagonal scaffold patterns exactly:

- **Port traits**: `#[async_trait]`, return `Result<T, DomainError>`, sealed error enums via `thiserror`. Pattern from `src/ports/greeting.rs` and `src/ports/notes.rs`.
- **Adapters**: Implement port trait, hold `PgPool` clone, async constructor. `sqlx::query!` for compile-time SQL verification. Pattern from `src/adapters/pg_notes.rs`.
- **HTTP handlers**: `async fn handler(State(state): State<AppState>, extractors...) -> Result<impl IntoResponse, ApiError>`. Domain errors converted to HTTP via `ApiError`'s `#[from]` impls. Pattern from `src/http/notes.rs`.
- **AppState**: Arc-wrapped adapters wired in `main.rs`, passed to router via `.with_state(state)`. Pattern from `src/main.rs`.
- **Migrations**: sqlx format (`NNNN_name.sql`), run at startup via `sqlx::migrate!()`. Pattern from `src/adapters/pg_notes.rs:connect()`.

New patterns introduced by this design:

- **Tower middleware for auth**: `middleware::from_fn_with_state` for HTTP Basic auth on the admin sub-router. No precedent in current scaffold; closest analogue is the `TraceLayer` in `http/mod.rs`.
- **Background task scheduling**: `tokio-cron-scheduler` jobs spawned as Tokio tasks in `main.rs`. Not present in current scaffold.
- **Multi-entry Vite build**: Two HTML entry points replacing the current single `index.html`. No precedent; Vite's `build.rollupOptions.input` handles it.
- **`src/ingestion/` and `src/enrichment/` modules**: Domain-adjacent orchestration logic that doesn't fit cleanly into `domain/`, `ports/`, or `adapters/`. These are pipeline coordination modules — pure Rust logic that calls port methods.

## Implementation Phases

<!-- START_PHASE_1 -->
### Phase 1: Bootstrap

**Goal:** Strip the example domain, establish the full DB schema, add required crates, and wire a compilable `AppState` skeleton so every subsequent phase has a solid foundation.

**Components:**
- Run `just clean-examples` to remove the Notes/Greeter example from `src/`, `tests/`, `migrations/`
- Port all 21 Flyway migrations from `kotlin-src/src/main/resources/db/migration/` to `migrations/` in sqlx format (`0001_init.sql` … `0021_add_source_id_to_event_sources.sql`), preserving all seed data (venues V8, sources V9, V11, V14–V20)
- Add to `Cargo.toml`: `scraper`, `reqwest` (with `json`, `rustls-tls` features), `tokio-cron-scheduler`, `axum-extra` (with `typed-header` feature); run `just bazel-repin`
- `src/config.rs` — `Config` struct with `Config::from_env() -> Result<Self, anyhow::Error>` mapping all env vars (database_url, bind_addr, admin credentials, ingestion/enrichment flags + cron schedules, connector API keys, MusicBrainz/Spotify settings, Discord webhook)
- `src/lib.rs` — add module stubs: `pub mod domain`, `pub mod ingestion`, `pub mod enrichment` alongside existing `pub mod adapters`, `pub mod ports`, `pub mod http`
- `src/http/mod.rs` — replace example `AppState` with new skeleton (`config: Arc<Config>` only for now); healthz still responds

**Dependencies:** None (first phase)

**Done when:** `just check` passes (fmt + clippy clean), `just test` passes, `just dev` starts and `/healthz` responds. All 21 migrations apply cleanly against a fresh database.

<!-- START_PHASE_2 -->
### Phase 2: Domain Types + Port Traits

**Goal:** Define all value types and port abstractions. No I/O. This phase is the type-checked specification of the entire domain.

**Components:**
- `src/domain/` — 7 submodules: `venue`, `artist`, `event`, `event_source`, `source`, `ingestion_run`, `featured_event`
  - ID newtypes wrapping `Uuid`: `VenueId`, `ArtistId`, `EventId`, `SourceId`, `FeaturedEventId`
  - Enums deriving `sqlx::Type`: `EventStatus`, `PriceTier`, `AgeRestriction`, `SourceType`, `EnrichmentStatus`, `IngestionRunStatus`, `EventType`, `EnrichmentSource`
  - Value structs: `Venue`, `Artist`, `Event`, `EventSource`, `Source`, `IngestionRun`, `FeaturedEvent`, `Page<T>`, `Pagination`
  - Ingestion types: `RawEvent`, `NormalizedEvent`, `DeduplicatedEvent`, `EventUpsertCommand`, `SourceAttribution`
  - Enrichment types: `EnrichmentResult` (canonical name, external ID, tags, image URL)
- `src/ports/` — 8 trait files (6 repository traits + `SourceConnector` + `ArtistEnricher`) as specified in Architecture section
- Domain error types: `RepoError`, `IngestionError`, `EnrichmentError` via `thiserror`

**Dependencies:** Phase 1 (module stubs exist, crates available)

**Done when:** `just check` passes with all domain types and port traits compiling. `just test` passes (property tests can be written for domain invariants: ID equality, enum round-trips, `RawEvent` field validation).

<!-- START_PHASE_3 -->
### Phase 3: PostgreSQL Adapters

**Goal:** Implement all database adapters. After this phase the full persistence layer is functional and integration-tested.

**Components:**
- `src/adapters/pg_venues.rs` — `PgVenueRepository` implementing `VenueRepository`: find by id/slug/neighborhood, paginated list, save
- `src/adapters/pg_artists.rs` — `PgArtistRepository` implementing `ArtistRepository`: find by id/name, paginated list, `claim_pending_batch` (SELECT FOR UPDATE SKIP LOCKED), save, reset orphaned IN_PROGRESS

- `src/http/mod.rs` — `AppState` updated with all six Arc-wrapped adapters; `main.rs` constructs them

**Dependencies:** Phase 2 (domain types + port traits)

**Done when:** Integration tests (`#[ignore]`, `just test-integration`) pass for: event upsert creates and updates correctly; slug deduplication prevents double-insert; artist `claim_pending_batch` is safe under concurrent access; enrichment state transitions (PENDING→IN_PROGRESS→DONE/FAILED/SKIPPED) persist correctly.

**Goal:** All HTTP endpoints functional. Public API queryable. Admin routes protected by Basic auth.

**Components:**

<!-- START_PHASE_5 -->
### Phase 5: Ingestion Pipeline

**Goal:** All source connectors implemented and tested. Ingestion pipeline runs end-to-end. Scheduler activates when `INGESTION_ENABLED=true`.

**Components:**
- `src/ingestion/normalization.rs` — `NormalizationService::normalize(Vec<RawEvent>) -> Result<Vec<NormalizedEvent>, IngestionError>`: title cleaning, slug generation (kebab-case from title + venue + date), time normalization
- `src/ingestion/deduplication.rs` — `DeduplicationService::deduplicate(Vec<NormalizedEvent>) -> Vec<DeduplicatedEvent>`: slug-keyed grouping, source attribution merging, confidence scoring
- `src/ingestion/orchestrator.rs` — `IngestionOrchestrator`: create run record → fetch → normalize → filter placeholder titles → deduplicate → upsert loop → update run + source health
- `src/ingestion/scheduler.rs` — `start_ingestion_scheduler(config, orchestrator, connectors)`: two `tokio-cron-scheduler` jobs (API cron + scraper cron from Config), gated on `INGESTION_ENABLED`

## Additional Considerations

**Migration numbering:** The Kotlin Flyway migrations use `V1__` through `V21__` naming. The sqlx equivalents use zero-padded 4-digit prefixes (`0001_` through `0021_`). The semantic content and order are preserved exactly — including all seed data migrations (venues, sources, deterministic UUIDs). The existing `migrations/0001_notes.sql` (example domain) is deleted by `just clean-examples` in Phase 1.

**`SELECT FOR UPDATE SKIP LOCKED` requirement:** The enrichment claim pattern requires PostgreSQL 9.5+. This is already guaranteed by the existing infrastructure (PostgreSQL is the only supported DB per the scaffolding).

**Connector HTTP client reuse:** All connectors share a single `reqwest::Client` instance (constructed in `main.rs` with a shared timeout + user-agent and passed through `AppState`). This avoids per-connector connection pool overhead.

**Implementation plan scoping:** This design has 7 phases. The `writing-plans` skill limit is 8, so a single implementation plan covers all phases.
